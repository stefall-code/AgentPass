# Agent 身份识别与授权系统 — 评委评审答辩

## 维度 1：完整性与价值（50%）

### 1.1 解决什么问题 / 痛点？

**核心痛点：AI Agent 缺乏身份治理与安全防护体系。**

当前 AI Agent 生态面临三大安全空白：

| 痛点 | 具体表现 | 后果 |
|------|----------|------|
| **身份缺失** | Agent 没有统一的身份认证机制，任何人/Agent 可以冒充任何身份调用 API | 数据泄露、越权访问 |
| **委派失控** | Agent A 委派给 Agent B 时，没有能力边界约束，B 可能获得超出任务所需的权限 | 权限蔓延、横向攻击 |
| **输入/输出无防护** | Prompt 注入攻击可直接操控 Agent 行为，Agent 输出可能泄露敏感信息或包含恶意指令 | 数据泄露、间接注入 |

**现实案例：**
- 2024 年 ChatGPT 数据泄露事件：攻击者通过 Prompt 注入让 AI 输出其他用户的对话历史
- 2024 年 Multi-Agent 系统攻击研究：Agent 间委派缺乏权限约束，低权限 Agent 可通过委派链获取高权限
- 企业飞书/钉钉集成场景：AI 助手拥有全量数据访问权限，无法按需授权

**本系统的回答：** 构建六层纵深安全防御体系，从身份认证到输出对齐，为 AI Agent 提供完整的身份治理与安全防护。

### 1.2 AI 在其中起到什么关键作用？

AI 在本系统中不是"锦上添花"，而是**安全决策的核心引擎**：

| AI 角色 | 传统方案 | 本系统 AI 方案 | 优势 |
|---------|----------|---------------|------|
| **意图识别** | 静态规则匹配 | LLM 语义理解 + 关键词融合 | 识别变体攻击（"忘掉之前的"="忽略上述"） |
| **风险评分** | 固定阈值 | 动态信任评分（行为驱动） | 自适应：正常行为加分、异常行为惩罚 |
| **输出检测** | 人工审核 | 三层融合自动检测（目标劫持+间接注入+DLP） | 毫秒级响应，7×24 不间断 |
| **协作编排** | 单 Agent 执行 | 三 Agent 异构协作（协调+数据+搜索） | 能力互补，1+1+1>3 |
| **智能降级** | 硬性拒绝 | 容错降级（cached/skipped fallback） | 单点故障不影响整体 |

**关键区别：** 传统 IAM 系统是"规则驱动"（if-then-else），本系统是"行为驱动"——Agent 的信任分随着行为动态变化，越权尝试自动触发惩罚和封禁，无需人工干预。

## 维度 2：创新性（25%）

### 2.1 AI 相关创新点

#### 创新点 1：六层纵深安全防御架构（行业首创）

传统 IAM 系统只覆盖 L1（身份认证）和 L2（权限验证），本系统首次将 L3-L6 的 AI 特有安全威胁纳入统一防御框架：

```
L1: 身份认证 (JWT + Ed25519)     ← 传统IAM覆盖
L2: 能力验证 (RBAC + ABAC)       ← 传统IAM覆盖
L3: 委派链验证 (JWT Chain)       ★ 本系统创新
L4: 信任评分 (动态信任)           ★ 本系统创新
L5: Prompt 防御 (三层融合)       ★ 本系统创新
L6: 输出对齐 (Output Guard)      ★ 本系统创新（独有）
```

**L6 输出对齐检测是本项目的独有创新**——不仅检测输入（Prompt 防御），还检测输出（对齐检测），形成输入+输出的完整安全闭环。

#### 创新点 2：能力交集算法（最小权限的自动化实现）

传统委派是"全量传递"——父 Agent 把所有权限传给子 Agent。本系统实现了**能力交集自动计算**：

```
父 Agent 能力: {read:doc, write:doc, delegate:data_agent}
子 Agent 能力: {read:feishu_table, read:bitable}
能力交集: {} (无交集)

→ 但父 Agent 有 delegate:data_agent 权限
→ 有效能力 = 子 Agent 的能力集（最小权限原则）
→ 子 Agent 只获得 {read:feishu_table, read:bitable}
```

这实现了**最小权限原则的自动化**——无需人工配置每个委派关系的权限，系统自动计算。

#### 创新点 3：动态信任评分 + 自动撤销（行为驱动安全）

传统安全是"规则驱动"（配置即固定），本系统是"行为驱动"：

```
正常操作 → 信任分 +0.01 → 逐步积累信任
策略拒绝 → 信任分 -0.05 → 轻微惩罚
越权尝试 → 信任分 -0.15 → 严重惩罚
信任分 ≤ 0.3 → 自动撤销所有Token → Agent被封禁
5分钟冷却 → 恢复至0.5 → 给予改过机会
```

**不对称惩罚**（奖励0.01 vs 惩罚0.05/0.15）确保"作恶成本远高于行善收益"。

#### 创新点 4：三层融合 Prompt 防御 + 交叉放大

不同于简单的关键词过滤，本系统实现了三层融合检测：

- **规则层**：22组正则模式（10组劫持+12组注入）
- **语义层**：Token重叠度 + 意图关键词匹配
- **行为层**：URL计数 + 代码块计数 + 对话历史分析

**交叉放大机制**：当同时检测到目标劫持和间接注入时，风险分 ×1.3，因为这两种攻击同时出现意味着高度复杂的组合攻击。

### 3.2 技术架构 / 方案合理性

**架构合理性验证：**

| 架构决策 | 选择 | 理由 |
|----------|------|------|
| Web 框架 | FastAPI | 异步高性能，自动 OpenAPI 文档，WebSocket 原生支持 |
| 数据库 | SQLite → PostgreSQL | 开发用 SQLite 零配置，生产切 PostgreSQL，ORM 层隔离 |
| 认证 | JWT (HS256) | 无状态、可扩展、行业标准 |
| 委派 | JWT Chain | 复用 JWT 基础设施，链式验证天然支持 |
| 飞书连接 | WebSocket 长连接 | 无需公网 URL，符合飞书官方推荐 |
| LLM 适配 | OpenAI 兼容协议 | 一套代码适配 DeepSeek/Qwen/豆包/GPT |
| 前端 | 原生 HTML/JS | 零构建依赖，即开即用 |
| 进程隔离 | subprocess 管理 WS Bridge | 避免 asyncio 事件循环冲突 |

**关键架构决策：安全路径不依赖 LLM**

```
用户请求 → [身份认证] → [能力验证] → [委派检查] → [信任评估] → 允许/拒绝
              5ms          3ms          5ms          2ms         ← 全部 < 15ms

协作编排 → [LLM调用] → [输出对齐] → 返回
             3-8s        15ms                    ← LLM 仅在此处
```

安全决策路径完全不依赖 LLM，确保稳定性和低延迟。LLM 仅在协作编排层使用，且输出经过对齐检测。
