## 一、核心产出（必填）

本周期交付的最满意成果是 **三Agent异构协作的完整安全闭环**——从"能跑通"到"可验证"的质变。

具体来说，之前三Agent协作（doc_agent → data_agent + external_agent → doc_agent 汇总）存在5个致命问题：外部搜索被拒绝、协作报告空白、飞书默认Mock模式、信任分状态残留、WebSocket刷屏。本周逐一定位根因并修复，最终实现了：

1. **独立Root Token签发机制** — 每步委派签发独立Token，解决了Token一次性消费导致的重放攻击误判。这是最核心的架构决策：将"一个Token走全程"改为"每步独立Token"，既保持了逐跳验证的安全性，又避免了Token复用冲突。

2. **三层输出对齐检测引擎**（`alignment.py`）— 行业首个面向Agent输出的安全检测引擎。目标劫持(40%) + 间接注入(35%) + DLP泄露(25%) 三层加权融合，劫持+注入同时触发时交叉放大×1.3，三级处置（block/warn/allow）。这不是简单的关键词过滤，而是将输入安全（Prompt Injection）和输出安全（Output Alignment）统一到一个框架中。

3. **能力交集算法**（`engine.py`）— 用数学实现了最小权限原则。三级计算策略：交集优先 → 通配符降级 → 委派权限兜底 → 拒绝。确保Agent委派后权限永远不超过自身声明。

这三个成果共同构成了系统的"安全三角"：**身份可信（Token链）+ 权限可控（能力交集）+ 输出安全（对齐检测）**，缺一不可。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

**卡点1：Root Token一次性消费问题（卡了最久，约2小时）**

- **现象**：三Agent协作时，step1（doc_agent → data_agent）成功，但step2（doc_agent → external_agent）总是返回"Replay attack detected"。
- **最初误判**：以为是external_agent的信任分不够，手动提升到0.9仍然失败。
- **破局**：在`engine.py`的`delegate()`方法中加日志，发现step1的`check()`调用了`mark_token_used(jti)`，把Root Token标记为已使用。step2再用同一Token时，`_check_replay()`检测到jti已存在，直接拒绝。
- **解决方案**：每步委派签发独立的Root Token。这不是简单的"多发一个Token"，而是架构层面的决策——每个委派步骤应该是独立的授权链，不共享状态。

**卡点2：Auto-Revoke状态残留（约1.5小时）**

- **现象**：运行自动撤销演示后，即使重置信任分到0.9，external_agent的所有请求仍被拒绝。
- **最初误判**：以为是信任分没有正确持久化，反复检查`_persist_trust_score()`。
- **破局**：在`check()`方法入口加断点，发现`is_agent_auto_revoked()`返回True——`AUTO_REVOKED_AGENTS`字典没有被清理。进一步发现数据库`TokenRevocationRow`表中也有残留记录。
- **解决方案**：`reset_trust_scores()`必须同时清理三层状态：内存字典（AUTO_REVOKED_AGENTS）、内存集合（REVOKED_AGENTS/REVOKED_TOKENS）、数据库表（TokenRevocationRow）。缺一不可。

**卡点3：WebSocket日志刷屏（约30分钟）**

- **现象**：调试控制台每秒打印3-5条"WebSocket closed"消息，无法看到其他日志。
- **破局**：改为指数退避重连（1s→2s→4s→...→30s max），只在首次断连打印日志，后续重连静默执行。

### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

**（1）"描述现象 → AI定位根因 → 生成修复代码 → 审查边界逻辑"的工作流**

这个模式在本周反复使用且效率极高。关键是"描述现象"要精确——不是"系统坏了"，而是"三Agent协作时step2返回Replay attack detected，step1正常"。AI能从精确的现象描述直接跳到根因，省去了大量逐行排查的时间。

**（2）独立Token签发模式**

每步委派签发独立Token的架构决策，可以复用到任何多步Agent协作场景。核心原则：**每个授权步骤应该是独立的授权链，不共享Token状态**。

**（3）三层融合检测框架**

目标劫持+间接注入+DLP的加权融合+交叉放大框架，可以扩展到更多检测层。下次只需定义新的检测层和权重，框架自动计算综合风险评分。

**（4）debug脚本模式**

`debug_collab.py`、`debug_revoke.py`等一次性调试脚本，虽然不会进入正式代码，但在定位问题时极其高效。下次遇到复杂状态问题时，先写一个最小复现脚本，比在完整系统中加日志快10倍。

## 四、项目介绍+Demo录屏演示+项目链接

### 项目简介

**AgentPass** 是一个 AI Agent 安全治理平台，解决多Agent协作中的三大安全空白：

1. **身份缺失** — Agent没有可验证的身份，任何人都可以冒充 → 我们用 A2A Token + 签名信任链解决
2. **委派失控** — Agent委派后权限无法约束，子Agent可能越权 → 我们用能力交集算法（最小权限原则）解决
3. **输入/输出无防护** — 只关注Prompt注入，忽略输出安全 → 我们用三层融合输出对齐检测解决

核心创新：**Prompt → Risk → Trust → Capability → Decision**，请求本身会改变权限结构。这不是传统IAM的"请求→权限→允许/拒绝"，而是"行为驱动权限动态收缩"。

### 技术架构

```
前端SPA (9页面) → FastAPI中间件栈 → 安全引擎层(6模块) → 协议适配层(A2A+MCP) → 数据持久层
```

六层纵深验证（L1身份→L2能力→L3委派链→L4行为→L5运行时→L6可观测性）在每个请求上实时执行，平均延迟<15ms。
