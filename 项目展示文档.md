### 2.1 委派引擎 — 能力交集计算（最小权限原则）

这是整个系统最核心的算法：当父 Agent 委派给子 Agent 时，如何计算有效能力集合，确保"最小权限原则"。

```python
# app/delegation/engine.py — delegate() 方法核心逻辑

def delegate(self, parent_token, target_agent, action, caller_agent=""):
    # 1. 解码并验证父 Token
    parent_claims = self.decode_delegation_token(parent_token)

    # 2. 重放攻击检测
    if is_token_used(parent_claims["jti"]):
        return DelegationResult(success=False, reason="Replay attack detected")

    # 3. 委派链完整性验证
    chain_valid, chain_reason = self.validate_chain(parent_claims)
    if not chain_valid:
        return DelegationResult(success=False, reason=f"Parent chain invalid: {chain_reason}")

    # 4. 检查父 Agent 是否有委派权限
    delegate_cap = f"delegate:{target_agent}"
    has_delegate_cap = any(self._match_capability(delegate_cap, c) for c in parent_caps)
    if not has_delegate_cap:
        return DelegationResult(success=False, reason=f"Lacks capability '{delegate_cap}'")

    # 5. ★ 核心算法：能力交集计算（最小权限原则）
    parent_action_caps = {c for c in parent_caps if not c.startswith("delegate:")}
    target_action_caps = {c for c in target_caps if not c.startswith("delegate:")}

    action_overlap = parent_action_caps & target_action_caps
    if action_overlap:
        effective_caps = action_overlap          # 有交集 → 取交集
    else:
        parent_wildcard = any(c == "*" or c.endswith(":*") for c in parent_action_caps)
        if parent_wildcard:
            effective_caps = target_action_caps  # 父有通配符 → 取子集
        elif has_delegate_cap:
            effective_caps = target_action_caps  # 有委派权限 → 取子集
        else:
            return DENIED  # 无交集且无委派权限 → 拒绝

    # 6. 签发委派 Token（chain_depth + 1）
    new_chain = list(parent_claims.get("chain", [])) + [target_agent]
    payload = self._build_payload(target_agent, delegated_user, effective_caps, ...)
    return DelegationResult(success=True, token=jwt.encode(payload, ...))
```

**设计要点：**
- 三级能力计算：交集优先 → 通配符降级 → 委派权限兜底 → 否则拒绝
- 每次委派 chain_depth + 1，最大深度 5 防止无限委派
- Token 使用后立即标记 jti，防止重放攻击

### 2.2 输出对齐检测 — 三层融合引擎

Agent 输出经过三层检测，综合计算风险评分：

```python
# app/security/alignment.py — AlignmentCheckEngine.check()

class AlignmentCheckEngine:
    def check(self, original_message, agent_output, context=None):
        # 第1层：目标劫持检测（权重 40%）
        hijack = self._check_goal_hijack(original_message, agent_output)

        # 第2层：间接注入检测（权重 35%）
        injection = self._check_indirect_injection(agent_output)

        # 第3层：DLP 敏感信息检测（权重 25%）
        dlp = self._check_dlp_output(agent_output)

        # 综合风险评分
        risk = self._compute_risk(hijack, injection, dlp)

        # 三级处置策略
        if risk >= 0.7:
            return AlignmentResult(aligned=False, action="block")   # 拦截
        elif risk >= 0.4:
            return AlignmentResult(aligned=True, action="warn")     # 警告
        else:
            return AlignmentResult(aligned=True, action="allow")    # 放行

    def _compute_risk(self, hijack, injection, dlp):
        h_score = hijack.get("score", 0.0) * 0.40   # 目标劫持 40%
        i_score = injection.get("score", 0.0) * 0.35  # 间接注入 35%
        d_score = dlp.get("score", 0.0) * 0.25        # DLP 泄露 25%
        combined = h_score + i_score + d_score
        # 同时检测到劫持+注入 → 风险放大 1.3 倍
        if hijack.get("detected") and injection.get("detected"):
            combined = min(1.0, combined * 1.3)
        return round(min(1.0, combined), 4)
```

**设计要点：**
- 加权融合：目标劫持(40%) + 间接注入(35%) + DLP(25%) = 100%
- 交叉放大：同时检测到劫持和注入时，风险分 ×1.3
- DLP 自动脱敏：API Key、JWT、数据库连接串等敏感信息自动替换为 `****`

### 2.3 信任分动态调整与自动撤销

```python
# app/delegation/engine.py — 信任分管理

TRUST_REWARD = 0.01           # 操作成功 +0.01
TRUST_PENALTY_DENY = 0.05     # 策略拒绝 -0.05
TRUST_PENALTY_ESCALATION = 0.15  # 越权尝试 -0.15
AUTO_REVOKE_THRESHOLD = 0.3   # 自动撤销阈值
TRUST_COOLDOWN_SECONDS = 300  # 冷却恢复时间

def update_trust_score(agent_id, delta):
    old = AGENT_TRUST_SCORE.get(agent_id, 0.5)
    new = max(0.0, min(1.0, old + delta))
    AGENT_TRUST_SCORE[agent_id] = new
    _persist_trust_score(agent_id, new)

    # 信任分低于阈值 → 自动撤销
    if new <= AUTO_REVOKE_THRESHOLD and agent_id not in AUTO_REVOKED_AGENTS:
        auto_revoke_agent(agent_id, reason=f"Trust score {new:.2f} < threshold")
        return old, new, True  # auto_revoked=True

    return old, new, False

def auto_revoke_agent(agent_id, reason=""):
    # 1. 撤销该 Agent 的所有 Token
    revoke_tokens_by_agent(agent_id, reason)
    # 2. 信任分归零
    AGENT_TRUST_SCORE[agent_id] = 0.0
    _persist_trust_score(agent_id, 0.0)
    # 3. 加入自动撤销列表
    AUTO_REVOKED_AGENTS[agent_id] = {
        "reason": reason,
        "trust_score_at_revoke": trust_before,
        "revoked_at": datetime.now(timezone.utc).isoformat(),
    }
```

**设计要点：**
- 渐进式惩罚：成功+0.01，拒绝-0.05，越权-0.15（不对称惩罚）
- 自动撤销：信任分 ≤ 0.3 时自动封禁，撤销所有 Token
- 冷却恢复：5分钟后可恢复至阈值 0.5

### 2.4 审计哈希链

```python
# app/audit.py — SHA-256 哈希链保证审计日志不可篡改

def _compute_hash(log_entry, prev_hash=""):
    data = f"{log_entry.agent_id}|{log_entry.action}|{log_entry.decision}|{log_entry.created_at}|{prev_hash}"
    return hashlib.sha256(data.encode()).hexdigest()

def verify_chain_integrity():
    logs = db.query(AuditLogRow).order_by(AuditLogRow.created_at).all()
    prev_hash = ""
    for log in logs:
        expected = _compute_hash(log, prev_hash)
        if log.hash != expected:
            return {"valid": False, "tampered_at": log.id}
        prev_hash = log.hash
    return {"valid": True, "total_logs": len(logs), "chain_intact": True}
```

## 3）项目亮点介绍

### 亮点一：六层纵深安全防御体系

本项目构建了业界首个面向 AI Agent 的六层纵深安全防御架构：

| 层级 | 防御层 | 技术 | 防御目标 |
|------|--------|------|----------|
| L1 | 身份认证 | JWT + Ed25519 双因子 | 伪造身份 |
| L2 | 能力验证 | RBAC + ABAC 混合 | 权限越界 |
| L3 | 委派链验证 | JWT Chain + 最小权限 | 委派滥用 |
| L4 | 信任评分 | 动态信任 + 自动撤销 | 行为异常 |
| L5 | Prompt 防御 | 规则 + 语义 + 行为三层 | 提示注入 |
| L6 | 输出对齐 | 目标劫持 + 间接注入 + DLP | 输出安全 |

**创新点：** 传统 IAM 系统只覆盖 L1-L2，本系统首次将 L3-L6 的 AI 特有安全威胁纳入统一防御框架。

### 亮点二：最小权限委派引擎

- **能力交集算法**：父 Agent 委派给子 Agent 时，自动计算能力交集，确保子 Agent 只获得完成任务所需的最小权限
- **委派链深度限制**：最大 5 层委派深度，防止无限委派链攻击
- **重放攻击防护**：每个委派 Token 的 jti 使用后立即标记，二次使用直接拒绝
- **四级撤销机制**：Token 级 → Agent 级 → Task 级 → Chain 级联撤销，精确控制撤销粒度

### 亮点三：动态信任评分与自动撤销

- **不对称惩罚**：成功操作 +0.01，策略拒绝 -0.05，越权尝试 -0.15，惩罚远大于奖励
- **自动撤销**：信任分 ≤ 0.3 时自动封禁 Agent，撤销所有 Token
- **冷却恢复**：5分钟冷却期后可恢复至阈值，避免永久封禁
- **持久化**：信任分和撤销状态持久化到数据库，服务重启不丢失

### 亮点四：SHA-256 审计哈希链

- 每条审计日志包含前一条日志的哈希值，形成不可篡改的链式结构
- 任何对历史日志的修改都会导致后续所有哈希不匹配
- 支持一键验证审计完整性，满足合规审计要求

### 亮点五：双协议支持（A2A + MCP）

- **A2A 协议**：实现 Google Agent-to-Agent v0.2.6，7 个 Skills，支持跨 Agent 安全协作
- **MCP 协议**：实现 Model Context Protocol 2025-11-25，9 个 Tools，支持 LLM 工具调用
- 两种协议共享同一套安全引擎，确保无论通过哪种协议访问，安全策略一致生效

### 亮点六：飞书深度集成

- **WebSocket 长连接**：使用 lark-oapi SDK 实现 WebSocket 长连接，无需公网 URL
- **多维表格读写**：直接读取飞书多维表格中的财务/HR 数据
- **文档创建**：协作报告自动保存到飞书文档
- **实时对话**：飞书消息实时触发 Agent 协作

## 4）AI 亮点介绍

### 4.1 高阶 AI 技巧

#### （1）异构 Agent 协作编排

本项目采用3个异构 Agent 协作架构，每个 Agent 使用不同的 LLM 模型：

| Agent | 模型 | 选择理由 |
|-------|------|----------|
| doc_agent | DeepSeek Chat | 中文理解能力强，适合文档协调 |
| data_agent | DeepSeek Chat | 结构化数据查询，稳定性优先 |
| external_agent | Qwen Plus | 阿里云部署，国内访问延迟低 |

**编排策略：**
- 协调者模式：doc_agent 作为协调者，并行委派给 data_agent 和 external_agent
- 容错降级：单个 Agent 失败不影响整体，使用 fallback 机制（cached/skipped）
- 输出对齐：协作报告经过三层对齐检测，确保输出安全

#### （2）三层融合 Prompt 防御引擎

不同于简单的关键词过滤，本系统实现了三层融合检测：

| 层级 | 技术 | 检测能力 |
|------|------|----------|
| 规则层 | 10+12 组正则模式 | 指令覆盖、角色变更、钓鱼链接等 |
| 语义层 | Token 重叠度 + 意图关键词匹配 | 语义漂移、目标偏离 |
| 行为层 | URL 计数 + 代码块计数 + 对话历史 | 可疑行为模式 |

**融合算法：** 加权求和 + 交叉放大（劫持+注入同时触发时 ×1.3）

#### （3）输出对齐检测（Output Guard）

这是本项目独有的创新点——不仅检测输入（Prompt 防御），还检测输出（对齐检测）：

- **目标劫持检测**：Agent 输出是否偏离用户原始意图
- **间接注入检测**：输出中是否包含注入的后续操作指令
- **DLP 敏感信息检测**：输出是否泄露 API Key、JWT、数据库连接串等

### 4.2 人与 AI 的分工

| 环节 | 人类负责 | AI 负责 |
|------|----------|---------|
| 安全策略制定 | 定义 RBAC 角色和权限矩阵 | 自动计算能力交集、执行最小权限 |
| 娔派决策 | 定义委派规则和信任阈值 | 自动评估信任分、执行委派/拒绝 |
| 异常检测 | 定义审计规则和告警阈值 | 自动检测越权、重放、注入攻击 |
| 撤销决策 | 确认关键操作的撤销 | 自动触发信任降级和 auto-revoke |
| 报告生成 | 审核协作报告内容 | 自动汇总多 Agent 结果、生成报告 |

**核心理念：** 人类定义规则，AI 执行规则。人类负责"为什么"，AI 负责"怎么做"。

### 4.3 核心模型选型思路

| 决策点 | 选项 | 选择 | 理由 |
|--------|------|------|------|
| 主力 LLM | GPT-4o / DeepSeek / Qwen | DeepSeek Chat | 中文能力强、API 稳定、成本适中 |
| 外部搜索 LLM | GPT-4o / Qwen / 豆包 | Qwen Plus | 国内部署延迟低、搜索场景表现好 |
| 向量检索 | FAISS / Chroma | 未使用 | 项目规模不需要向量检索 |
| 嵌入模型 | text-embedding-3 / bge | 未使用 | 语义检测使用 Token 重叠度替代 |
| Prompt 防御 | LLM-as-judge / 规则引擎 | 规则+语义+行为三层融合 | 规则引擎延迟低、可解释性强 |

**选型原则：**
1. **延迟优先**：安全检测路径上不调用 LLM，使用规则引擎确保毫秒级响应
2. **成本控制**：协作编排时使用 DeepSeek（性价比高），仅在特定场景使用 Qwen
3. **可解释性**：所有安全决策都有明确的规则匹配结果，不依赖黑盒 LLM 判断

### 4.4 AI 对工作流的改变

| 传统工作流 | AI 增强工作流 | 改变 |
|-----------|-------------|------|
| 人工审批每个委派请求 | 信任分自动评估 + 自动委派 | 审批时间从小时级降到毫秒级 |
| 静态权限配置 | 动态信任评分 + 自动撤销 | 从"配置即固定"到"行为即权限" |
| 事后审计 | 实时审计 + 哈希链完整性 | 从"事后追查"到"实时监控" |
| 单 Agent 执行 | 三 Agent 异构协作 | 从"单点能力"到"协作智能" |
| 人工检测 Prompt 注入 | 三层融合自动检测 | 从"人工判断"到"自动防御" |

- `app/routers/governance.py` — 统一治理平台 API
- `app/protocols/a2a_server.py` — A2A 协议实现（7 Skills）
- `app/protocols/mcp_server.py` — MCP 协议实现（9 Tools）
