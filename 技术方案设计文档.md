### 1.2 异构 Agent 架构

系统采用3个异构 Agent 协作架构，每个 Agent 具有不同的能力边界和信任等级：

| Agent | 角色 | 信任分 | 能力范围 | LLM 模型 |
|-------|------|--------|----------|----------|
| doc_agent | 协调者 | 0.90 | 34个能力（文档读写+委派权限） | DeepSeek Chat |
| data_agent | 数据读取 | 0.95 | 12个能力（飞书多维表格只读） | DeepSeek Chat |
| external_agent | 外部搜索 | 0.70 | 1个能力（read:web） | Qwen Plus |

### 1.3 六层安全验证架构

```
Layer 1: 身份认证 (JWT + Ed25519)
    ↓
Layer 2: 能力验证 (RBAC + ABAC)
    ↓
Layer 3: 委派链验证 (JWT Chain + 最小权限)
    ↓
Layer 4: 信任评分 (动态信任 + 自动撤销)
    ↓
Layer 5: Prompt 防御 (规则 + 语义 + 行为)
    ↓
Layer 6: 输出对齐 (目标劫持 + 间接注入 + DLP)
```

## 2. Access Token 设计

### 2.1 Token 体系

系统采用双 Token 机制：

| Token 类型 | 有效期 | 用途 | 算法 |
|------------|--------|------|------|
| Access Token | 60分钟 | API 访问凭证 | HS256 |
| Refresh Token | 7天 | 刷新 Access Token | HS256 |
| Delegation Token | 60分钟 | 委派链中的临时凭证 | HS256 |

### 2.2 Access Token Payload

```json
{
  "sub": "agent:doc_agent",
  "role": "coordinator",
  "jti": "uuid-v4",
  "iat": 1715000000,
  "exp": 1715003600,
  "usage_limit": 1000,
  "type": "access",
  "bound_ip": "192.168.1.100"
}
```

### 2.3 Delegation Token Payload

```json
{
  "sub": "agent:data_agent",
  "role": "data_reader",
  "jti": "uuid-v4",
  "iat": 1715000000,
  "exp": 1715003600,
  "parent_jti": "parent-token-jti",
  "chain_depth": 1,
  "delegated_user": "user:demo_judge",
  "capabilities": ["read:feishu_table:finance"],
  "type": "delegation"
}
```

### 2.4 Token 生命周期

```
签发(issue) → 使用(check) → 刷新(refresh) → 过期(expire)
                ↓                ↓
           标记已用          撤销(revoke)
                ↓
         重放检测(mark_used)
```

### 2.5 Token 安全机制

| 机制 | 说明 |
|------|------|
| IP 绑定 | Token 可绑定客户端 IP，防止 Token 被盗用 |
| 用量限制 | 默认 1000 次，超出自动失效 |
| 重放检测 | 每个 Token 的 jti 使用后标记，防止重放攻击 |
| 四级撤销 | Token 级 / Agent 级 / Task 级 / Chain 级联撤销 |
| 过期检查 | exp 字段自动校验 |

## 5. 委派引擎核心逻辑

### 5.1 委派流程

```
1. issue_root_token(agent_id, delegated_user, capabilities)
   → 签发根 Token，chain_depth=0

2. delegate(parent_token, target_agent, action, caller_agent)
   → 验证父 Token 有效性
   → 计算能力交集（最小权限原则）
   → 检查 chain_depth <= MAX_CHAIN_DEPTH(5)
   → 检查目标 Agent 信任分 >= TRUST_THRESHOLD(0.5)
   → 检查目标 Agent 未被 auto-revoke
   → 签发委派 Token，chain_depth+1
   → 标记父 Token 为已使用（防重放）

3. check(token, action)
   → 解码 Token，验证签名和过期
   → 检查 Token 未被撤销
   → 检查 action 在 capabilities 范围内
   → 检查 Agent 信任分
   → 更新信任分（成功+0.01，失败-0.05/-0.15）
   → 触发 auto-revoke 检查
```

### 5.2 能力交集计算

```python
parent_action_caps = set(parent_capabilities) & set(ALL_ACTIONS)
target_action_caps = set(CAPABILITY_AGENTS[target_agent]["capabilities"]) & set(ALL_ACTIONS)

action_overlap = parent_action_caps & target_action_caps
if action_overlap:
    effective_caps = action_overlap
elif parent_has_delegate_permission:
    effective_caps = target_action_caps
elif parent_has_wildcard:
    effective_caps = target_action_caps
else:
    return DENIED  # 最小权限原则
```

### 5.3 信任分动态调整

| 事件 | 调整值 |
|------|--------|
| 操作成功 | +0.01 |
| 策略拒绝 | -0.05 |
| 越权尝试 | -0.15 |
| 信任分 ≤ 0.3 | 自动撤销（auto-revoke） |
| 冷却恢复（5分钟） | 恢复至 0.5 |

### 5.4 审计链

每个操作记录审计日志，使用 SHA-256 哈希链保证完整性：

```
log[0].hash = SHA256(log[0].data)
log[n].hash = SHA256(log[n].data + log[n-1].hash)
```

验证时重新计算哈希链，任何篡改都会导致验证失败。
